《Git学习指南》学习笔记

    日期：2018/10/20 作者：Shallwe

    分类：学习笔记

　　在大二的时候接触了Git即版本控制系统，在学习的时候可能用的机会不多，但在实际工作中和团队协作中它能起到很大的作用。虽然入门会稍微难一点，但是功能很强大很值得学。我分享的是自己在阅读《Git学习指南》和其他资料的过程中记录的入门笔记。有一些部分我没有深究就跳过了。当时写的时候可能会比较粗糙望见谅。如有错误也望指点。

【第1章 基本概念】

　　Git是分布式版本控制系统。
　　分支的创建与合并。

【第2章 入门】

　　修改Git for Windows的HOME环境变量
　　　　打开cmd，输入sysdm.cpl，高级->环境变量
　　初次使用要配置机器
　　git config --global user.name "..."
　　git config --global user.email "..."
　　创建版本库 git init
　　首次提交
　　　　1. git add .　　git add a.txt （版本中添加）　　git rm a.txt （版本中删除）
　　　　2. git commit --message "新的版本..."
　　检查状态　git status
　　显示日志　git log　（--oneline 更简洁）（--gragh 图形化）
　　　　　　　　　　（--stat 显示被修改的文件）（-n 3 最近3次）
　　协作功能
　　克隆版本库 git clone /pro1 /pro1_clone
　　git pull
　　　　将原版本库中的新提交传递给它的克隆体（在克隆体目录进行）
　　　　pull（拉回）从原版本库中取回新的修改
　　　　如果发生了冲突，则需要手动挑选文件
　　git pull /pro1_clone master （在原版本库中拉回克隆体的修改）（在原版本目录进行）
　　共享版本库
　　　　创建裸版本库　git clone --bare /pro1 /pro1_bare.git
　　　　裸版本库用来充当开发者们push的汇聚点，用来交换提交
　　git push
　　　　向指定版本库传送提交　git push /pro1_bare.git master
　　　　让克隆版本也得到该修改用pull

【第3章 提交究竟是什么】

　　每一次提交都带有提交的散列值 40个字符组成的编码
　　在Git中，回复到某一版本通常称之为检出（checkout）操作
　　git diff (散列值)　　 比较两次提交之间的差异（--stat 显示文件修改）
　　git diff --word-diff　按单词显示所做的修改

【第4章 多次提交】

　　工作区 -- 暂存区 -- 版本库
　　git diff --staged　显示暂存区与提交的不同
　　git diff　　　　　 显示暂存区与工作区的不同
　　git reset HEAD .　 重置暂存区，撤回修改
　　使用 .gitignore 忽略非版本控制文件
　　　　在文件中写好路径即可忽略
　　git stash　将工作区和暂存区的修改保存在储藏栈（stash stack）的缓存区中
　　git stash pop　恢复被储藏的修改
　　git stash list　列出储藏栈

【第5章 版本库】

　　讲的是底层结构（管道层）

【第6章 分支】

　　git branch new_branch1　　 创建新分支
　　git branch --list　　　　　查看所有分支
　　git checkout new_branch1　 切换到分支
　　git branch -d new_branch1　删除分支（-D强制删除）
　　checkout被拒绝（由于工作区还存在着一些修改）时解决办法
　　　　1. 提交修改即可　　　git commit --all
　　　　2. 放弃修改强制切换　git checkout --force new_branch1
　　　　3. 储藏修改即可　　　git stash　（git stash pop恢复）
　　恢复被删除分支
　　　　1. git reflog　查找要恢复分支的散列值
　　　　2. git branch new_branch1 d765a1e　敲入散列值恢复

【第7章 合并分支】

　　git merge new_branch1　合并分支（在要合并的分支上如master）
　　手动合并解决冲突
　　　　法一：手动编辑受影响文件至不冲突
　　　　法二：git checkout --theirs tests/　（--theirs只采用别人的版本文件）
　　　　　　　git checkout --ours tests/　　（--ours　只采用自己的版本文件）
　　　　最后再 git add . 和 git commit
　　快进合并
　　　　一条线上，一个分支没有动，而另一个分支一只更新，要把落后的分支快进合并
　　　　git checkout behind_branch
　　　　git merge front_branch

【第8章 通过变基净化历史】

　　git rebase master　可以将带有很多小分支和分岔的历史改造成一部较为平滑的线性历史
　　移植分支等，了解一下

【第9章 版本库间的交换】

　　有很多协议来访问版本库，如file,http,ftp,ssh等
　　如 git clone file:///Users/git_book.git
　　给版本库起名字，使用git remote add
　　　　git remote add MyClone file:///tmp/git_book_clone.git
　　去掉名字，使用git remote rm
　　　　git remote rm MyClone
　　查看推送提交路径　git remote --verbose
　　从另一个版本库中获取提交（将克隆版本库传送到本地版本库）
　　　　git fetch MyClone
　　将本地提交传送到远程版本库，使用push
　　　　git push clone feature-a
　　　　（推送至clone所指向的远程版本库并更新分支指针）
　　显示远程跟踪分支
　　　　git branch -r
　　从远程分支中分岔出一个本地分支
　　　　git checkout -b feature-b clone/feature-b
　　利用本地分支操作别处的版本库
　　　　git fetch clone feature-b:my-feature-b　（别处分支名:本地分支名）
　　pull = fetch + merge
　　提交本地test分支，作为远程的master分支
　　　　git push origin test:master
　　删除远程版本库中的分支
　　　　git push clone :feature-a

【第10章 版本标签】

　　为一个分支创建标签（-m为注释）
　　　　git tag 1.1.0.0 master "Freshly built"
　　推送标签
　　　　git push origin 1.1.0.0
　　在日志中查看标签
　　　　git log --decorate

【其它】

　　git gui　图形界面
　　新建远程仓库，将代码上传到Github的示例操作：

echo "# New Repo" >> README.md
git init
git add README.md
git commit -m "first commit"
git remote add origin https://github.com/github-id/repo-id.git
git push -u origin master

【Issues】

　　git pull 提示 fatal: refusing to merge unrelated histories （新建远程仓库常遇到）
　　　　两个根本不相干的 git 库，一个是本地库，一个是远端库，然后本地要去推送到远端，远端觉得这个本地库跟自己不相干，所以告知无法合并。
　　　　解决方法一：从远端库拉下来代码，本地要加入的代码放到远端库下载到本地的库，再上传。
　　　　解决方法二：不相干的分支强行合并。 git pull origin master --allow-unrelated-histories
